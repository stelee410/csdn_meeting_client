# 消息推送机制内部化改造 - 任务总结

## 一、改造概述

将原有的CSDN外部消息推送服务改造为系统内部推送，通过数据库存储站内信消息，并提供查询接口供前端拉取。

### 核心变更
- **原方式**：调用CSDN消息中心API发送IM站内信
- **新方式**：消息存储到本地数据库 `t_user_message`，用户通过API拉取

---

## 二、数据库变更

### 2.1 新增表

**V23__create_user_message_table.sql**

| 表名 | 说明 |
|------|------|
| `t_user_message` | 用户消息表（站内信），存储会议发布、报名审核等通知 |

### 2.2 表结构

```sql
t_user_message (
    message_id      -- 消息业务ID（MSG前缀）
    user_id         -- 接收用户ID
    message_type    -- 1-会议发布 2-报名通过 3-报名拒绝
    title           -- 消息标题
    content         -- 消息内容
    biz_id          -- 关联业务ID（会议ID）
    biz_type        -- 业务类型（MEETING/REGISTRATION）
    extra_data      -- 扩展数据（JSON格式）
    is_read         -- 是否已读
    read_time       -- 阅读时间
    is_deleted      -- 是否删除（软删除）
    created_at      -- 创建时间
)
```

## 四、API接口清单

### 消息中心接口

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/v1/messages` | GET | 获取消息列表（分页） |
| `/api/v1/messages/unread-count` | GET | 获取未读消息数 |
| `/api/v1/messages/{id}/read` | POST | 标记消息已读 |
| `/api/v1/messages/read-all` | POST | 标记全部已读 |
| `/api/v1/messages/{id}` | DELETE | 删除消息（软删除） |

```
┌─────────────────────────────────────────────┐

│    云计算技术沙龙                            │
│    昨天                                    │
├─────────────────────────────────────────────┤
│  [加载更多...]                               │
└─────────────────────────────────────────────┘
```

### 6.2 交互流程
1. **进入页面**：调用`GET /api/v1/messages`获取消息列表
2. **未读提醒**：调用`GET /api/v1/messages/unread-count`获取未读数，建议定时轮询（如每30秒）
3. **点击消息**：调用`POST /api/v1/messages/{id}/read`标记已读，跳转会议详情页
4. **全部已读**：点击"全部已读"按钮调用`POST /api/v1/messages/read-all`
5. **删除消息**：点击删除按钮调用`DELETE /api/v1/messages/{id}`

## 八、风险与注意事项

1. **消息历史**：CSDN历史消息无法迁移，改造后新消息才存储到本地表
2. **消息拉取**：改为被动拉取模式，需前端配合实现消息中心页面
3. **数据量**：随着消息增长，需考虑归档策略（如3个月前的消息自动归档）

---

## 九、后续优化建议

1. **WebSocket推送**：增加WebSocket长连接，实现消息实时推送
2. **消息模板**：增加消息模板管理，支持运营人员配置消息内容
3. **消息归档**：定期将历史消息归档到冷存储，减轻主库压力
